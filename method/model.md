# The map model — schema v2 (JSON source)

From schema v2 on, the committed source of truth of a coyodex map is a **structured JSON model**:
`.coyodex/project-map.json`. The markdown map and the HTML diagram are **generated views** of it,
written next to it (`.coyodex/project-map.md`, `.coyodex/project-map.html`) and regenerated by the
tooling — never hand-edited. This inverts schema v1 ("markdown is source, `model.json` is an
ephemeral parse"): the parse **is** the source now, made explicit.

Why: every parsing-fragility bug class (escaping, table shape, split tables, block micro-formats,
hand-merge malformations) existed only because the source was semi-structured markdown needing a
bespoke parser. The model is read by the standard library, unambiguously; `validate` starts with
schema validation; every tool reads fields. Rationale: the maintainer's data-model proposal
(Decision 1). The catch it names holds here too: a structured source only works together with
**structured authoring** — build agents return structured rows and `coyodex assemble` serializes
the model; the LLM never hand-authors the stored file.

What did NOT change: the ID scheme, the element kinds, the edge-verb vocabulary and its
"verbs may prioritize, never gate" principle, the single-source membership rules, and everything
"derived, not duplicated" — all of [schema-v1.md](schema-v1.md)'s semantic contract carries over.
Only the storage format and the authoring path changed. Schema-v1 markdown remains readable by the
one-time converter (`coyodex convert`), not by the tools.

## The document

One JSON object. Key order below is the canonical serialization order (the serializer always emits
it, so diffs are stable); arrays keep authored order. `null` means "not stated". All text cells are
strings that MAY contain inline markdown (a `[label](path#Lnnn)` link, backticks) — a literal `|`
needs no escaping (the markdown-view generator escapes it when rendering tables).

```
{ "format": "coyodex-map/2",
  "title":  "<project display name>",
  "goal":   "<T0 prose — the problem the project solves and for whom>",
  "commit": "<short sha>", "committed": "<YYYY-MM-DD>", "built": "<YYYY-MM-DD HH:MM>",

  "roles":       [ { "name", "kind": "human|service", "wants", "drives" } ],
  "glossary":    [ { "term", "meaning", "where" } ],
  "use_cases":   [ { "id": "UCn", "name", "actor", "trigger_outcome" } ],
  "golden_path": [ { "id": "GPn", "title", "uc": "UCn", "why": "<prerequisite or null>" } ],

  "subsystems":  [ { "id": "Sn",  "name", "purpose", "parent": "Sn|null", "anchor", "confidence" } ],
  "components":  [ { "id": "Cn",  "name", "subsystem": "Sn|null", "purpose",
                     "entry_point": "<md link|null>", "depends_on": "<derived summary text>",
                     "anchor": "<canonical source anchor|null>", "confidence", "extra": {…} } ],
  "deps":        [ { "id": "Dn",  "name", "kind": "<closed vocab|null>", "type", "used_for",
                     "where_configured", "confidence", "deployment_linked": false, "extra": {…} } ],
  "run_commands":  [ { "action", "command", "source" } ],                       // T3
  "entry_points":  [ { "kind", "trigger", "entity": "<md link>", "component": "Cn" } ],  // T4

  "subdomains":  [ { "id": "SDn", "name", "purpose", "parent": "SDn|null", "anchor", "confidence" } ],
  "entities":    [ { "id": "En",  "name", "store", "meaning", "subdomain": "SDn|null",
                     "source": "<path#Lnnn|null>",
                     "fields":    [ { "name", "type", "markers": ["PK", "FK→En", "[]", …] } ],
                     "relations": [ { "verb", "target": "En", "src_card", "dst_card",
                                      "display", "how" } ] } ],
  "non_entity_types": [ { "name", "source", "why" } ],

  "flows":       [ { "uc": "UCn", "title",
                     "steps": [ { "n", "src", "dst", "phrase", "note" } ] } ],  // T6
  "edges":       [ { "src", "verb", "dst", "why", "where": "<call-site md link|null>" } ],

  "deployment":     [ { "unit", "runs_on", "exposed_as", "config_source" } ],
  "observability":  [ { "signal", "where_emitted", "where_viewed", "alerts" } ],
  "security":       [ { "surface", "who", "check": "<md link to the auth check>", "risk" } ],
  "config":         [ { "key", "purpose", "default", "per_env" } ],

  "tests_note":  "<the 'Tests run for this table?' honesty line>",
  "tests":       [ { "target", "tested", "tests", "gap", "confidence" } ],
  "extras":      [ { "heading", "body": "<verbatim markdown>" } ] }
```

Semantics carried over from schema v1, stated on the fields:

- **IDs** are `prefix + digits` only; each element's `id` must match its array's prefix. Every
  reference (a `subsystem`/`parent` pointer, an edge endpoint, a flow-step endpoint, a GP step's
  `uc`, a relation `target`, an ID mentioned in any text) must resolve to a defined element —
  `coyodex validate` checks it, exactly as before.
- **Membership is single-source on the child**: `components[].subsystem`, `subsystems[].parent`,
  `entities[].subdomain`, `subdomains[].parent`. Member lists, inter-group edges, and the
  subsystem→subdomain bridge stay **derived, never stored**.
- **`edges` is ONE project-wide backbone list** (`C↔C`, `C↔D`, `C→E`; `E↔E` stays on the cards).
  Schema v1's two authored edge tables (T1 backbone + Relationships) are unified here; the markdown
  view renders the one list. Duplicated authored rows are preserved as authored (the graph views
  de-duplicate by `(src, verb, dst)`, as the v1 parser did).
- **`flows[].steps`**: an endpoint is an element ID or a Role display name (an actor step);
  element↔element steps are pure references to the backbone edge (its verb + why render the step).
- **`entities[].relations`** are authored on the source card only; `verb` uses the canonical
  structural verbs (`contains`/`has`/`isA`) or a free association verb; cardinality is a pair
  (`src_card`/`dst_card`, both or neither); `how` is the plain-text note for field-less relations.
- **`deps[].kind`** is the closed Context vocabulary (`datastore`/`messaging`/`service`/`platform`/
  `framework`/`library`); when `null` it is inferred from `type`, as before.

## New fields (schema v2 only — each closes a measured v1 gap)

- **`components[].anchor`** — the element's **canonical source anchor** (where the component
  *lives*: its home file or directory). v1 had no such field; tools took "the component's file"
  from the first link in its row (usually the entry point), a heuristic that mis-describes
  umbrella components. The L2 grounding claims now describe a component by this anchor **plus its
  member entry points** (every `entry_points` row naming the component) — so a skeptic sees an
  umbrella component ("Event stream — in-process + Redis") through ALL its member entry points,
  never one arbitrary file. This is the fix for the false-refutation class found at the Phase-1
  boundary.
- **Deps are described as external systems, never as code files.** A dep endpoint in an L2 claim
  reads "`D4 = Google OAuth (service: Google OAuth 2.0 endpoints)`" — its `kind` + `type` — not a
  source anchor; a component calling the real external system must never be refuted because the
  dep was anchored at some local wrapper module.
- **`deps[].deployment_linked`** — `true` marks a dep wired at **deployment level only** (nginx,
  Vector, Kibana…): it genuinely has no code call site, so the "every T2 dep needs an incoming
  edge" nudge does not fire for it — and the method still forbids inventing a `C→D` edge for it.
- **`non_entity_types`** — an explicit plumbing marker: named types living in the domain dirs that
  are deliberately NOT entities (repositories, providers, error hierarchies). The under-harvest
  coverage check excludes them by this list (the v1 name-suffix heuristic is kept as a fallback
  and still applies).

The edge `verb` is deliberately **unchanged** (one authored word; prioritize-never-gate). Verb
facets / code-verified categories are a documented later direction (Phase 4), not schema v2.

## Views are generated, never edited

- `coyodex render project-map.json project-map.md` writes the markdown view (canonical section
  order, template-shaped tables, a "GENERATED — do not edit" notice). It is committed alongside the
  JSON for readable diffs; `coyodex validate` warns when the committed view is stale (differs from
  the regenerated one).
- `coyodex render project-map.json project-map.html` writes the interactive HTML viewer, as before.
- The serializer is **deterministic**: the same model always produces byte-identical JSON (fixed
  key order, `indent=2`), so the committed JSON diffs cleanly.

## Authoring: structured rows → `coyodex assemble`

Build agents return **structured rows** (JSON fragments — each a partial model: a subset of the
top-level arrays, plus at most one `header` fragment for title/goal/pin). The lead saves each
fragment verbatim and runs:

```
coyodex assemble <fragment.json>… --out .coyodex/
```

`assemble` validates every fragment against the schema (one bad fragment fails alone, with its
path named), merges by ID (a duplicate ID across fragments is an error, never a silent overwrite),
and serializes the canonical `project-map.json`. Validity of the stored file is guaranteed by the
serializer, not by the LLM. Then the usual invariant runs unchanged: `validate → audit → render`.
